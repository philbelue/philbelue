Key insight
Customer Admin and OEM Engineer both see all five sites — but their roles give completely different capabilities once inside. Groups = scope. Roles = capability. These are independent axes. Never conflate them.
Three rules that keep your IXON environment scalable as customers and machines grow.
01
Keep roles broad and reusable
One "Vendor Technician" role serves all vendors. Scope is managed by group assignment — not by creating new roles per customer.
02
Use groups to control scope
Model your hierarchy in groups by site, division, or machine line. Adding a user to a group instantly grants visibility — no role edits needed.
03
Apply least privilege via access categories
Assign only what the role needs — Vendor Technician gets VNC/HTTP, not Alarms. Don't grant all categories by default.
Common anti-patterns to avoid
✕Creating a new role per customer instead of using groups
✕Assigning all access categories to every role by default
✕Skipping groups and assigning devices individually to users
✕Using Platform Admin role for standard customer logins
Six steps to configure your IXON environment correctly the first time. Complete in order — each step depends on the one before it.
1
Define your group hierarchy
Model your environment before touching the platform. Sketch how your customers, sites, and machines are organized — groups should mirror this hierarchy.
Customer   └── Site         └── Production line                 └── Machine type
Groups control what users can see. Get this right first.
2
Create reusable roles
Roles represent job functions — not specific customers or sites. One role per function type, reused across all groups.
Platform AdminEngineerCustomer AdminVendor TechnicianOperatorRead Only
Avoid: "Nestle Admin" or "TOMCO Engineer" — keep roles generic.
3
Define access categories
Access categories map to the specific platform resources each role will need.
VPN AccessAlarmsDashboardsMonitoringVNC / HTTP
4
Assign access categories to roles
Link each role to only the categories it needs. Apply least privilege.

6
Validate access
Never assume access is working as intended — test before handing off to the customer.
Use a test account or incognito browser
Log in as a user with a restricted role and confirm they see only their assigned groups and can only access their assigned categories. Fix before going live.
